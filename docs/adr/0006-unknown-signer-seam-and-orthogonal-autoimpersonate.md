# Unknown-signer seam at the single broadcast choke point; `autoImpersonate` and `onUnknownSigner` are orthogonal

When a privileged call (proxy upgrade, admin action, raw tx, deploy, value transfer) targets an account the deployer key cannot sign for — a Safe multisig, a hardware wallet, an air-gapped key — rocketh surfaces the exact tx to execute out-of-band via a first-class `UnknownSignerError` (`@rocketh/core`), thrown at the SINGLE `broadcastTransaction` choke point in `packages/rocketh/src/environment`. All txs funnel through that function, so one check is enough — hardhat-deploy v1 repeated the same check in five places; rocketh does it once. The error carries `{from, to?, data?, value?, contract?: {name, method, args}}`, with `contract` populated only when the call originated from an `execute` so the printed message can name the intended function.

`autoImpersonate` and `onUnknownSigner` are deliberately kept **orthogonal**. `autoImpersonate` is a NODE CAPABILITY switch (impersonate named unsignable accounts IF the node supports it) and runs BEFORE the seam; if it resolves the account, the seam never fires. `onUnknownSigner: 'throw' | 'auto'` is the policy for accounts that remain unsignable AFTER impersonation. `onUnknownSigner` intentionally has no `'impersonate'` value — collapsing the two would conflate "can this node fake signatures?" (a capability) with "what should happen when we genuinely can't sign?" (a policy). Default is `'auto'`; while no interactive resolver exists, `'auto'` degrades to `'throw'` so CI never prompts or hangs. `catchUnknownSigner` forces the `'throw'` policy for its wrapped action via an environment-level push/pop stack (dynamic-scope, safe because rocketh executes deploy scripts sequentially / single-await); this same mechanism will later back a per-call override without any further seam changes.

Nothing is persisted by this mechanism — no unsigned-tx file, no state — mirroring hardhat-deploy v1 exactly. Idempotency is on-chain-state-driven: the user executes the deferred tx on their Safe, re-runs the idempotent script, and the on-chain state check skips the completed step. A persisted batch, if ever built, belongs downstream in a consumer package (see `explore-unknown-signer-adapters`), not in the core seam — keeping it out of core preserves the "one choke point, zero side effects" property that makes the seam auditable.
