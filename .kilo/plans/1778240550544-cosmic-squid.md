# Plan: Tag Tracking and Selective Reset Features

## Overview

This plan addresses two related features for the rocketh deployment system:

1. **Tag Tracking**: Record each script's own declared tags on its deployments
2. **Dependency Tracking**: Record each script's declared dependencies (as tags) on its deployments
3. **Selective Reset**: Use tag and dependency information to safely reset deployments

---

## Feature 1: Track Tags and Dependencies on Deployments

### Problem
Currently, when a deployment script runs:
1. The tags declared on that script are not recorded in the deployment file
2. The dependencies (as tags) declared on that script are not recorded

This makes it impossible to:
- Track which tags were responsible for a deployment
- Understand script dependencies for selective reset
- Detect when resetting a deployment would break other deployments

### Solution

#### 1.1 Update Deployment Type
**File**: `packages/rocketh-core/src/types.ts`

Add optional `tags` and `dependencies` fields to the `Deployment` type:

**Note**: The `dependencies` field tracks the script's declared dependency tags (from `scriptModule.dependencies`), not runtime deployment accesses.

## Feature 2: Selective Reset with Script Dependency Awareness

### Problem
The `--reset` flag currently deletes ALL deployments for an environment. Users want to reset only specific deployments, but this can break other deployments that depend on them.

**Example issue**: 
- `core` script has tag `core`
- `feature1` script has tag `feature1`, depends on `core`
- `feature2` script has tag `feature2`, depends on `core`

Running `--reset --tags feature1` would:
1. Reset `core` (dependency of `feature1`)
2. Reset `feature1`
3. Leave `feature2` pointing to the old `core` deployment → **broken state**

### Solution: Reverse Script Dependency Resolution

When resetting tags, we need to:
1. Find all scripts that will be reset (via tag + script dependencies)
2. Find all scripts that depend on those scripts (reverse lookup)
3. Also reset deployments from those dependent scripts
4. Show the user the full scope and require confirmation

#### 2.1 Reuse `--tags` for Selective Reset
**Decision**: Instead of adding a new `--reset-tags` option, reuse the existing `--tags` option.

**Behavior**:
- `--tags X` without `--reset`: Execute scripts with tag X (current behavior)
- `--reset` without `--tags`: Delete all deployments (current behavior)
- `--reset --tags X`: Delete deployments from scripts with tag X, their dependencies, AND scripts that depend on them (new behavior)

#### 2.2 Selective Reset with Reverse Script Dependency Resolution

#### 2.3 Implement getScriptsAndTagsToDelete Function
**File**: `packages/rocketh/src/executor/index.ts`

```typescript
async function getScriptsAndTagsToDelete(
  tags: string[],
  scriptIdBags: {[tag: string]: string[]},
  scriptModuleById: {[id: string]: DeployScriptModule},
): Promise<{scriptIdsToDelete: Set<string>; deploymentTagsToDelete: Set<string>}> {
  const scriptIdsToDelete = new Set<string>();
  const deploymentTagsToDelete = new Set<string>();
  const visitedScriptIds = new Set<string>();
  
  // Step 1: Resolve script dependencies recursively (forward)
  function resolveScriptDependencies(scriptId: string) {
    if (visitedScriptIds.has(scriptId)) return;
    visitedScriptIds.add(scriptId);
    scriptIdsToDelete.add(scriptId);
    
    const scriptModule = scriptModuleById[scriptId];
    if (scriptModule?.tags) {
      for (const tag of scriptModule.tags) {
        deploymentTagsToDelete.add(tag);
      }
    }
    
    // Recursively resolve script dependencies
    if (scriptModule?.dependencies) {
      for (const depTag of scriptModule.dependencies) {
        const depScriptIds = scriptIdBags[depTag] || [];
        for (const depScriptId of depScriptIds) {
          resolveScriptDependencies(depScriptId);
        }
      }
    }
  }
  
  // Start from scripts with the specified tags
  for (const tag of tags) {
    const scriptIds = scriptIdBags[tag] || [];
    for (const scriptId of scriptIds) {
      resolveScriptDependencies(scriptId);
    }
  }
  
  // Step 2: Find scripts that depend on the scripts we're deleting (reverse lookup)
  const visitedForReverse = new Set<string>();
  
  function findDependentScripts(scriptId: string) {
    if (visitedForReverse.has(scriptId)) return;
    visitedForReverse.add(scriptId);
    
    // Find all scripts that have this script's tags as dependencies
    const scriptModule = scriptModuleById[scriptId];
    const scriptTags = scriptModule?.tags || [];
    
    for (const scriptTag of scriptTags) {
      // Find all scripts that depend on this tag
      for (const [otherScriptId, otherScriptModule] of Object.entries(scriptModuleById)) {
        if (otherScriptModule?.dependencies?.includes(scriptTag)) {
          if (!scriptIdsToDelete.has(otherScriptId)) {
            scriptIdsToDelete.add(otherScriptId);
            if (otherScriptModule.tags) {
              for (const tag of otherScriptModule.tags) {
                deploymentTagsToDelete.add(tag);
              }
            }
            // Recursively find dependents of this script
            findDependentScripts(otherScriptId);
          }
        }
      }
    }
  }
  
  // Start from scripts we're already deleting
  for (const scriptId of scriptIdsToDelete) {
    findDependentScripts(scriptId);
  }
  
  return {scriptIdsToDelete, deploymentTagsToDelete};
}
```

#### 2.4 Delete Deployments by Tags
**File**: `packages/rocketh/src/executor/index.ts`

```typescript
async function deleteDeploymentsByTags(
  deploymentStore: DeploymentStore,
  deploymentsFolder: string,
  environmentName: string,
  deploymentTagsToDelete: Set<string>,
): Promise<{deletedDeployments: string[]; orphanedDeployments: string[]}> {
  const fileNames = await deploymentStore.listFiles(deploymentsFolder, environmentName);
  const deletedDeployments: string[] = [];
  const orphanedDeployments: string[] = [];
  
  for (const fileName of fileNames) {
    if (fileName.endsWith('.json') && fileName !== '.migrations.json' && fileName !== '.pending_transactions.json') {
      const deployment = JSON.parse(await deploymentStore.readFile(deploymentsFolder, environmentName, fileName));
      const deploymentTags = deployment.tags || [];
      
      // Check if any deployment tag matches tags to delete
      const hasMatchingTag = deploymentTags.some(tag => deploymentTagsToDelete.has(tag));
      const name = fileName.slice(0, fileName.length - 5);
      
      if (hasMatchingTag) {
        await deploymentStore.deleteFile(deploymentsFolder, environmentName, fileName);
        deletedDeployments.push(name);
      }
    }
  }
  
  // Clean up migrations for deleted deployments
  // Clean up pending transactions for deleted deployments
  
  return {deletedDeployments, orphanedDeployments};
}
```

## Testing Strategy

### Tag and Dependency Tracking Tests
- Verify that deployments created by scripts with tags have those tags recorded
- Verify that deployments created by scripts with dependencies have those dependencies recorded
- Verify that deployments created by dependency scripts have their own tags (not the triggering tag)
- Verify that multiple tags are all recorded (e.g., script with tags `['X', 'Y']`)

### Selective Reset Tests
- Test that `--reset` without tags still deletes all deployments
- Test that `--reset --tags X` deletes deployments with tag `X` and their script dependencies
- Test that scripts depending on reset scripts are also included (reverse dependency)
- Test edge cases: deployments with no tags, deployments with multiple tags
- Test that `--tags X` without `--reset` still works normally

---

## Backward Compatibility

- The `tags` and `dependencies` fields on `Deployment` are optional, so existing deployments without them will continue to work
- Selective reset is opt-in via `--reset --tags` combination
- Default `--reset` behavior (without `--tags`) remains unchanged (deletes all)
- Default `--tags` behavior (without `--reset`) remains unchanged (executes scripts)
- For existing deployments without `tags` field, they won't be deleted by selective reset (safe default, but user might need full reset)

## Open Questions (Resolved)

1. **Should we track the "executing tag" separately from "dependency tags"?**
   - **Decision**: No, only record the script's own declared tags on deployments
   - Rationale: Deployment content should be independent of how it was triggered

2. **How do we handle migrations file during selective reset?**
   - **Decision**: Yes, delete `.migrations.json` entries for deleted deployments

3. **What about pending transactions during selective reset?**
   - **Decision**: Delete `.pending_transactions.json` entries for deployments being reset

4. **How to pass current script tags to the environment?**
   - **Decision**: Add setter functions to the `Environment` interface that the executor calls before executing each script

5. **How to handle deployments that depend on reset deployments?**
   - **Decision**: Use reverse script dependency lookup to find scripts that depend on scripts being reset, and include their deployments in the reset
   - This ensures we don't leave orphaned deployments
