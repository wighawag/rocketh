# Migrate the docs site to Astro Starlight

Decision for now: **stay on the unified VitePress site.** This note captures a real, working evaluation of Astro Starlight as a future replacement, so it can be picked up later without redoing the exploration.

## Context

The docs site is a single VitePress build: the rocketh docs (root `index.md`, `README.md` -> `introduction.md`, `documentation.md`) plus the hardhat-deploy docs (`hardhat-deploy/` at repo root, exposed under `/hardhat-deploy/`), one config in `.vitepress/config.mts`, local search. This unification (folding hardhat-deploy in, dropping the old two-site build + `cp` dance) is already done and shipped.

Starlight was prototyped against the **real** rocketh + hardhat-deploy content and produced a working 25-page site. This note is the durable record of that; the throwaway prototype itself lived outside the repo and can be recreated from the steps below.

## Why consider Starlight (the wins)

- **Unified search for free** (Pagefind indexes every page). VitePress local search is per-config; Starlight searches the whole site including the hardhat-deploy section out of the box.
- **Autogenerated sidebar** from a directory (`autogenerate: { directory }`), which removes most of the hand-maintained sidebar we carry in the VitePress config today (the whole hardhat-deploy/how-to tree came in automatically).
- **Docs-native features** shipped by default: "On this page" TOC, heading anchors/deep-links, dark-mode toggle, prev/next, breadcrumbs, components (`<Tabs>`, `<CardGrid>`, asides).
- **Fits the stack** (Astro/Svelte are already used elsewhere by wighawag), and has headroom for i18n and versioned docs.
- The home hero maps cleanly: VitePress `layout: home` -> Starlight `template: splash` + `<CardGrid>`.

## Migration cost (measured, not guessed)

These are the concrete things that had to be done to get real content building, i.e. the actual work if we do this for real:

1. **Frontmatter `title` on every doc.** Starlight requires `title:` in frontmatter; our markdown uses `# H1` headings instead. A ~20-line script derives `title` from the first H1, strips inline-code backticks, removes the H1 (Starlight renders the title itself), and writes the frontmatter. ~25 files, one-time, scriptable. (Script is in the "Recreate the prototype" section.)
2. **Convert VitePress containers to components.** `::: code-group` (and any `::: tip/warning`) render **literally** in Starlight; they must become `<Tabs>`/`<TabItem>` and Starlight asides. This is the main manual content edit.
3. **Normalize internal links.** The hardhat-deploy markdown uses relative `./foo.md` links (these already work in VitePress and mostly resolve in Starlight, but Starlight prefers extensionless slugs; should be normalized for correctness). Also repo-relative links in the rocketh README (`./public/...`, `./LICENSE`, `./packages/...`) need handling or the page should be trimmed for the site.
4. **New dependency/upgrade surface.** Astro + Starlight + `sharp` (native, needs build-script approval) is a fresh, large dep tree, i.e. a new Dependabot surface to maintain, and Starlight has config API churn across minors (e.g. the `autogenerate` group syntax changed in v0.39: must be `{ label, items: [{ autogenerate: { directory } }] }`, not `label` on the autogenerate object).
5. Minor: set `site` in config (sitemap), and add a custom 404 slug (Starlight warns `Entry docs -> 404 was not found`).

## Verdict

Starlight is the best "proper docs" upgrade with the least custom code (search/TOC/anchors/autogenerated nav are free). The cost is a **bounded, one-time** content migration (frontmatter titles + container-to-component conversion + link normalization) plus owning a new dep/upgrade surface. It does **not** simplify over the already-unified VitePress site by itself; do it for the features, not for simplification. If the current VitePress site is "good enough", staying is the right default (which is what we're doing).

## Recreate the prototype (when we want to build it for real)

```bash
# scaffold (outside the repo, throwaway)
pnpm create astro@latest starlight -- --template starlight --no-install --no-git --skip-houston --yes
cd starlight

# bring in real content
mkdir -p src/content/docs/hardhat-deploy
cp -r <repo>/hardhat-deploy/documentation/* src/content/docs/hardhat-deploy/
cp <repo>/documentation.md src/content/docs/documentation.md
cp <repo>/README.md      src/content/docs/rocketh-readme.md
rm -rf src/content/docs/guides src/content/docs/reference
```

Frontmatter-title script (`add-frontmatter.mjs`, run once over `src/content/docs`):

```js
import {readdirSync, readFileSync, writeFileSync, statSync} from 'node:fs';
import {join} from 'node:path';
function walk(dir){for(const e of readdirSync(dir)){const p=join(dir,e);statSync(p).isDirectory()?walk(p):p.endsWith('.md')&&process(p);}}
function process(p){
  let src=readFileSync(p,'utf8');
  if(src.startsWith('---'))return;
  const m=src.match(/^#\s+(.+)$/m);
  let title=(m?m[1]:p.split('/').pop().replace(/\.md$/,'')).replace(/`/g,'').trim();
  if(m)src=src.replace(/^#\s+.+$\n?/m,'');
  writeFileSync(p,`---\ntitle: ${JSON.stringify(title)}\n---\n\n`+src.replace(/^\n+/,''));
}
walk('src/content/docs');
```

Sidebar (`astro.config.mjs`): a `rocketh` group with `documentation` + `rocketh-readme`, and a `hardhat-deploy` group listing the ordered top pages plus a nested `{ label: 'How-To Guides', items: [{ autogenerate: { directory: 'hardhat-deploy/how-to' } }] }`.

Home (`src/content/docs/index.mdx`): `template: splash` with `hero.actions` (Documentation -> `/documentation/`, hardhat-deploy -> `/hardhat-deploy/introduction/`, GitHub) and a `<CardGrid>` of the three feature cards (Declarative / Replicable / Modular), mirroring the current VitePress home.

Then `pnpm install` (approve `sharp` + `esbuild` build scripts) and `pnpm build`. Expect a working 25-page site with Pagefind search.
